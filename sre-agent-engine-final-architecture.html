Finalized Architecture · v2.0

SRE Toil Automation Agent Engine

A Strategist → Dynamic Runbook → Sub-Agent architecture. The Strategist Agent ingests signals from all MCP tool surfaces, uses domain reasoning to generate a situation-specific runbook, compiles it to YAML, and spawns specialized sub-agents to execute each task — with human approval gates before any remediation.

Research-backed by: AWS Bedrock supervisor-agent pattern, Microsoft's LLM incident management research, mcp-agent orchestration framework, and the Rootly/incident.io dynamic runbook model.
01 Core Architecture — 4 Layers
L1 · UI

Operations Dashboard

React + Vite

Incident Feed

Live stream of triggered runs, their state (diagnosing → planning → executing → verifying), and drill-down into each step's output.

Generated Runbook Viewer

Shows the YAML runbook the Strategist generated for this incident. Editable before execution. Diffable against previous runs.

Approval Queue

Critical path. Shows proposed remediation with full diagnosis context. Approve/reject/modify. Audit trail per action.

MCP Registry

Register MCP servers, health-check connections, browse discovered tools. Each tool tagged as read or write.

Why React + Vite over Next.js or HTMX? This is an internal SRE tool, not a public-facing app — SEO and SSR are irrelevant. Vite gives sub-second HMR for rapid iteration. React is the safest bet for hiring and component ecosystem (Monaco editor for YAML, xterm.js for log streaming). HTMX was considered but rejected because the approval queue needs real-time WebSocket state that HTMX handles poorly.
L2 · API

FastAPI Backend

Python 3.12 · async

POST /incidents

Receive alert webhooks or manual triggers. Validates context (app, env, severity), creates run record, pushes to engine.

WS /runs/{id}/stream

WebSocket stream of execution events. Each step emits: started → tool_called → output_received → decision_made.

POST /approvals/{id}

Approve or reject a gated remediation step. Logs who approved, when, and the full context they saw. Resumes engine.

GET /runbooks/{id}

Fetch the generated YAML runbook for any run. Supports version history — every edit before execution is tracked.

CRUD /mcp-servers

Register, update, delete MCP server configs. On register, auto-discovers tools and caches their schemas.

GET /runs

Paginated run history. Filter by app, env, status, time range. Full audit trail for compliance.

Why FastAPI over Flask/Django? The engine runs concurrent MCP calls and streams results via WebSocket — both require first-class async. FastAPI is natively async (Starlette + uvicorn), has Pydantic-based validation (which we already need for runbook schemas), and auto-generates OpenAPI docs. Django's ORM is sync-first and adds unnecessary weight. Flask + SocketIO is possible but requires more glue code for the same result.
LangGraph / LangChain: Adds framework coupling, graph DSL complexity, and a large dependency tree for something we can express in ~300 lines of pure Python async. The engine's control flow is a simple state machine, not a graph. Temporal: Excellent for durable execution at scale (used by Stripe, Netflix), but adds an infrastructure dependency (Temporal server + worker cluster). For MVP, Python asyncio with SQLite-backed state checkpointing gives us durability without ops overhead. Temporal is the clear v2 choice when you need multi-node execution.
L3 · ENGINE

Agent Engine — The Brain

Claude API + MCP SDK

This is the core differentiator. Three-phase architecture: Diagnose → Plan → Execute.

Strategist Agent

Phase 1 — Diagnose: Receives incident context. Calls read-only MCP tools across ALL registered servers (Splunk, Dynatrace, SLO tracker, etc.) to gather comprehensive situation data. Runs in parallel where possible.

Phase 2 — Plan: Feeds all tool outputs into Claude with domain-specific system prompt. Claude generates a structured runbook: ordered steps, tool calls, decision gates, and remediation actions. Output is validated against Pydantic schema and serialized to YAML.

Sub-Agent Spawner

Phase 3 — Execute: Parses the generated YAML runbook. For each step, spawns a lightweight sub-agent — a bounded async task with its own MCP client, tool scope, and timeout. Sub-agents run sequentially by default, parallel when the runbook specifies parallel: true.

Each sub-agent type: DiagnosticAgent (read-only MCP calls), DecisionAgent (LLM reasoning at branch points), RemediationAgent (write MCP calls — always gated).

Context Accumulator

Maintains a rolling context window across all sub-agent outputs. Summarizes long tool outputs (Splunk queries can return 50KB+) to stay within token budget. Uses a 3-tier strategy: raw output (if <2K tokens), extractive summary (2K–8K), abstractive summary (8K+).

Approval Gate

Intercepts any step where type: remediation or approval_required: true. Pauses execution, pushes full diagnosis context + proposed action to the approval queue. Resumes only on human approval. Timeout configurable (default: 30 min → auto-escalate).

Why Strategist generates the runbook dynamically instead of following pre-written YAML? Three reasons: (1) Failure modes are combinatorial — a pre-written runbook for "high error rate" can't anticipate that today's cause is an OOM from a memory leak in a new deployment vs. an upstream DNS timeout vs. a certificate expiry. The LLM adapts the plan to what it actually sees in the data. (2) Tool surfaces evolve — when you add a new MCP server (say, PagerDuty), the Strategist automatically incorporates it into its plans without rewriting every playbook. (3) Institutional knowledge capture — the Strategist's system prompt encodes your team's SRE domain expertise (escalation rules, SLO thresholds, known architectural weaknesses), which is easier to maintain as a prompt than as branching YAML. The generated YAML is still the execution artifact — auditable, diffable, reproducible — but it's generated per-incident, not per-category.
Why not fully autonomous agent (ReAct loop)? A ReAct agent that decides its own next action at every step is powerful but unpredictable. In SRE, unpredictable = dangerous. The Strategist generates the FULL plan upfront, which means: (a) a human can review the plan before execution starts, (b) the plan is deterministic once approved, (c) token cost is bounded (one planning call + N small execution calls, not an open-ended loop). This is the "plan-then-execute" pattern from AWS Bedrock's supervisor model, not the "step-by-step" ReAct pattern.

Why MCP over direct API clients? (1) Tool discovery — the Strategist's Claude call receives tool schemas from ALL registered MCP servers dynamically. Add a new server, and the Strategist can use its tools immediately without code changes. (2) Uniform interface — every tool call returns structured JSON regardless of whether the backend is REST, GraphQL, or gRPC. The engine doesn't need per-service adapters. (3) Security boundary — MCP's tool-level permissions let you tag tools as read/write at the protocol level. The engine enforces: write tools → approval gate. This is cleaner than trying to classify arbitrary API calls. (4) Ecosystem momentum — Claude, Cursor, Windsurf, and every major AI tool now speaks MCP. Your investment compounds.
The Core Invariant: All read operations (diagnosis, metrics, logs, SLO checks) run autonomously. All write operations (remediation, scaling, restarts, notifications) require explicit human approval. The LLM proposes; a human disposes. This is non-negotiable for production SRE at JPMC's risk tolerance.
02 End-to-End Execution Flow

Walk through a real incident: "CDS API Gateway error rate spike to 12% in production". Three phases: Diagnose → Plan → Execute.

PHASE 1 — DIAGNOSE
TRIGGER

1. Alert Received

Dynatrace webhook fires → POST /incidents. Context injected: app=cds-api-gw, env=prod, severity=CRITICAL, error_rate=12.3%. Run record created in SQLite.

MCP · PARALLEL

2. Strategist Gathers Intelligence

The Strategist fires parallel read-only MCP calls to all registered servers. No LLM call yet — this is pure data gathering.

→ dynatrace://get_metrics

error_rate, p99_latency, throughput for last 30m

→ splunk://search

ERROR logs from cds-api-gw, top messages, stack traces

→ slo_tracker://get_burn_rate

Current error budget, 1h/6h/24h burn rates

→ aiops://get_correlated_alerts

Other alerts in the same time window across dependent services

→ aiops://get_change_events

Recent deployments, config changes, infra events

Why parallel, not sequential? These are independent read queries. Running 5 MCP calls in parallel via asyncio.gather() takes ~max(latencies) instead of ~sum(latencies). For typical Splunk (3s) + Dynatrace (1s) + others (1s), that's 3s vs 7s. In incident response, those 4 seconds matter.
PHASE 2 — PLAN
STRATEGIST · LLM

3. Claude Generates Dynamic Runbook

All MCP outputs fed into a single Claude Sonnet call with the SRE Domain System Prompt. Claude receives:

• Full tool outputs from step 2 (summarized if over token budget)
• Available MCP tools (auto-discovered schemas for all registered servers)
• Domain rules (SLO thresholds, escalation policies, known architectural patterns)
• Historical context (last 5 runs for this app, if available)

Claude returns a structured runbook via tool_use — not free-text. The output is a JSON object conforming to the RunbookSchema Pydantic model, which is then serialized to YAML.

Why a single planning call instead of iterative reasoning? Token economics. One Sonnet call with ~8K context + ~2K output ≈ $0.04. An iterative ReAct loop averaging 6 turns ≈ $0.15+. At 100 incidents/day, that's $4/day vs $15/day — and the plan-then-execute model gives you a reviewable artifact BEFORE anything runs. This is the pattern AWS Bedrock's supervisor uses: plan upfront, execute deterministically.
Why Claude Sonnet, not Opus? For the planning call, Sonnet is sufficient — the task is structured output generation from provided data, not open-ended reasoning. Sonnet is 5x cheaper and 3x faster. Reserve Opus for edge cases where Sonnet's plan fails validation (automatic fallback).
OPTIONAL GATE

4. Human Reviews Plan (Configurable)

For critical/production environments, the generated runbook is pushed to the UI for human review BEFORE execution begins. The SRE can approve as-is, edit the YAML, or reject and manually investigate. For staging/dev, this gate can be skipped via config.

PHASE 3 — EXECUTE
SUB-AGENTS

5. Sub-Agents Spawned from Runbook

The executor parses the YAML and spawns a sub-agent for each step. Example generated plan for this incident:

DiagnosticAgent #1

Deep log analysis. Run targeted Splunk query for OOM/heap patterns in the stack traces identified in Phase 1.

→ splunk://search (read)

DiagnosticAgent #2

Upstream check. Query Dynatrace topology for CDS gateway's upstream dependencies. Check their error rates.

→ dynatrace://get_topology (read)

DecisionAgent

Root cause synthesis. LLM call with deep-dive outputs. Determines: OOM? Upstream? Config? Returns structured diagnosis.

→ claude (structured output)

RemediationAgent ⚠

Rolling restart. Restart CDS gateway pods one at a time. GATED — requires human approval.

→ k8s://restart_deployment (write)

Why sub-agents instead of a monolithic executor? (1) Isolation — each sub-agent has its own timeout, retry policy, and error boundary. A failing Splunk query doesn't abort the whole run. (2) Parallelism — DiagnosticAgent #1 and #2 above run in parallel since they're independent. The runbook schema supports parallel_group declarations. (3) Typed behavior — DiagnosticAgents can only call read tools. RemediationAgents can call write tools but are always gated. This is enforced in code, not by LLM compliance. (4) Observability — each sub-agent emits its own event stream. The UI shows a per-agent trace, making debugging straightforward.
APPROVAL GATE

6. Human Approves Remediation

RemediationAgent pauses. The approval queue shows: full diagnosis chain, root cause hypothesis (with confidence), proposed action (rolling restart), risk assessment, rollback plan. SRE reviews and approves.

✓ Approve — execute as planned ✎ Modify — edit params then approve ✗ Reject — escalate to human investigation
EXECUTE

7. Remediation + Verify

On approval: RemediationAgent executes k8s://restart_deployment. Logs everything. Then spawns a final VerificationAgent that waits 5 minutes and re-checks error rate via Dynatrace. If recovered → close with summary. If not → escalate with full context attached.

End-to-end timing estimate: Phase 1 (parallel MCP): ~3s. Phase 2 (Claude planning): ~4s. Phase 3 (sub-agent execution): ~variable. Total diagnosis-to-plan: ~7 seconds vs. the typical 10-15 minutes of human context-switching across Splunk, Dynatrace, and Confluence runbooks. That's 100x faster to first hypothesis.
03 Generated Runbook Schema

This is an example of what the Strategist generates for the CDS gateway incident. The schema is defined as a Pydantic model; Claude outputs conforming JSON via tool_use, which is then serialized to YAML for human readability and audit.

generated_runbook_cds_gw_2026-03-20T14:23:00.yaml
# Auto-generated by Strategist Agent # Incident: CDS API Gateway error rate spike (12.3%) # Generated: 2026-03-20T14:23:07Z metadata: run_id: "run_a7f3c9e2" app: "cds-api-gw" environment: "prod" trigger: "dynatrace_alert_DT-4891" strategist_model: "claude-sonnet-4-20250514" confidence: 0.82 diagnosis_summary: | Error rate spiked to 12.3% (baseline 0.4%) at 14:18 UTC. Splunk logs show java.lang.OutOfMemoryError in 73% of error traces. A deployment (build-4891) was pushed at 14:15 UTC — 3 min before spike. SLO burn rate at 14.2x (critical). Return: confirmed (bool), confidence (float), recommended_action (enum). output_schema: confirmed: bool confidence: float recommended_action: "enum[rollback, restart, scale_up, escalate]" reasoning: string branches: rollback: "execute_rollback" restart: "execute_restart" escalate: "escalate_to_oncall" - id: "execute_rollback" type: "remediation" # ← triggers approval gate agent: "RemediationAgent" approval_required: true # ← ENFORCED, cannot be false for write ops description: "Rollback to previous build" tool: server: "kubernetes" method: "rollback_deployment" params: deployment: "cds-api-gw" namespace: "prod" to_revision: "previous" rollback_plan: "Re-deploy build-4891 if rollback causes regression" risk: "medium" - id: "verify_recovery" type: "diagnostic" agent: "DiagnosticAgent" depends_on: ["execute_rollback"] wait_before: "5m" # wait for rollback to propagate description: "Verify error rate has recovered" tool: server: "dynatrace" method: "get_metrics" params: entity: "cds-api-gw" metrics: ["error_rate", "p99_latency"] timeframe: "last-5m" success_criteria: error_rate_below: 1.0 # percent on_failure: "escalate_to_oncall" - id: "escalate_to_oncall" type: "remediation" agent: "RemediationAgent" approval_required: false # notification is safe to auto-send description: "Page on-call with full context" tool: server: "incident_mgmt" method: "page_oncall" params: service: "cds-api-gw" severity: "P1" context: "{{full_run_summary}}"
Key schema design decisions: (1) parallel_group — steps in the same group run concurrently. The executor uses asyncio.gather() for same-group steps. (2) depends_on — explicit DAG edges. The executor won't start a step until all dependencies are resolved. (3) output_schema on decision steps — enforced via Claude's tool_use. The LLM MUST return typed, structured output. No free-text parsing. (4) branches — deterministic routing. The DecisionAgent's output maps directly to the next step ID. (5) approval_required: true is auto-set by the engine for any step where type: remediation AND the tool is tagged as write. Even if the Strategist generates false, the engine overrides it. Defense in depth.
04 Architecture Decision Log

Every architectural choice, stress-tested. What was chosen, what was rejected, and why.

DECISION
CHOSEN → WHY
REJECTED → WHY NOT
Agent Pattern
Supervisor (plan-then-execute) — Strategist generates full plan, sub-agents execute deterministically. Auditable, reviewable, cost-bounded. Matches AWS Bedrock's proven supervisor pattern.
ReAct loop — Each step decides next action. Unpredictable cost, hard to audit, can hallucinate tool calls. Dangerous for production SRE. Peer-to-peer — No central coordinator. Harder to debug, no single plan to review.
Runbook Origin
LLM-generated per incident — Adapts to actual signals. Handles combinatorial failure modes. New tools auto-incorporated. Domain knowledge lives in system prompt (easier to maintain than YAML branches).
Pre-written static YAML — Can't anticipate novel failures. Every new failure pattern = new runbook. Maintenance burden grows linearly with incident categories.
Orchestration Framework
Pure Python asyncio — ~300 lines of executor code. No framework lock-in. Full control over MCP client lifecycle, token budgets, approval gates. Easy to debug with standard Python tools.
LangGraph — Graph DSL adds abstraction without capability gain for this use case. Large dep tree. CrewAI — Opinionated about agent roles, poor MCP support. Temporal — Excellent but adds infra (server + workers). Clear v2 upgrade path.
LLM Model
Claude Sonnet 4 for planning + decisions. Fast (2-4s), cheap ($3/1M input), excellent tool_use. Structured output via tool schemas eliminates parsing. Opus fallback for when Sonnet plan fails validation.
GPT-4o — Strong but more expensive, less reliable structured output. Local models — Insufficient reasoning for root cause analysis. Opus-only — 5x cost for marginal planning quality improvement.
Tool Interface
MCP (Model Context Protocol) — Universal tool discovery. Read/write tagging at protocol level. Claude native support. Ecosystem growing fast. Your existing Splunk/Dynatrace servers already speak MCP.
Direct API clients — Per-service adapters, no unified discovery, manual read/write classification. OpenAPI — Good for REST but doesn't cover non-REST tools. Custom tool format — Reinventing MCP with worse ecosystem.
State Persistence
SQLite + SQLModel — Zero ops. Single file. Handles 100 concurrent runs easily. Full ACID. Perfect for MVP. Migration to Postgres is a config change, not a rewrite.
PostgreSQL — Adds container/service to manage. Overkill for <1000 runs/day MVP. Redis — Volatile. Incidents need durable audit trails. File-based — No query capability for run history.
Sub-Agent Design
Typed agents (Diagnostic/Decision/Remediation) — Each type has hardcoded permissions. DiagnosticAgent literally cannot call write tools (enforced in code, not by LLM). Separation of concerns. Easy to test in isolation.
Generic agent — One agent type with runtime permissions. Relies on LLM to respect boundaries. One hallucination = unauthorized write. Unacceptable risk.
Human-in-the-Loop
2-tier gating — (1) Optional plan review before execution. (2) Mandatory approval before any write operation. Defense in depth. Configurable per environment (prod=both gates, staging=write-only, dev=none).
Fully autonomous — Your risk team will kill this project day one. Approve every step — Approval fatigue. SREs will stop using it within a week.
Token Cost Control
Context Accumulator with 3-tier summarization — Raw (<2K tokens), extractive (2K-8K), abstractive (8K+). Budget of 12K tokens per planning call. At $3/1M tokens, this caps cost at ~$0.04/incident for planning.
Send everything raw — A single Splunk search can return 50KB (~15K tokens). 5 parallel calls = 75K tokens = $0.22 just for input. At 100 incidents/day = $22/day just for context. Unsustainable.
Frontend
React + Vite + Tailwind — Fast dev, WebSocket-native for real-time streaming, Monaco editor for YAML editing. Internal tool = no SSR needed.
Next.js — SSR/SEO irrelevant for internal tool. HTMX — Poor WebSocket story for real-time execution streaming. Streamlit — Fast prototype but can't handle approval queue complexity or custom YAML editor.

07 What's NOT in MVP (v2 Roadmap)

Confidence-based auto-approve

If same runbook pattern approved 20+ times → offer auto-approve with 30s countdown. Prevents approval fatigue.

Temporal migration

Move executor to Temporal workflows for durable, multi-node execution. Critical for scaling past 100 incidents/day.

Runbook learning loop

Feed resolved incident data back into strategist prompt. RAG over historical runbooks to improve future plans.

Visual runbook builder

Drag-and-drop DAG editor for manually creating/editing runbooks. Nice UX layer over the YAML schema.

Slack/Teams integration

Approve remediation from Slack. Get diagnosis summaries in incident channels. ChatOps-native.

RBAC

Role-based approval flows. Junior SRE can approve restarts, senior SRE required for rollbacks, manager for scaling.
