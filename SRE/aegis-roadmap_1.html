01 · The Frame

What we shipped, what's missing, where it has to land.

The demo proved the architecture works: Strategist diagnoses, the Dynamic Runbook composes a plan, Sub-Agents execute through MCP, the Verify Loop confirms recovery. That's the 20% that earns the right to build the other 80%. The 80% is everything that turns a working pattern into something a Tier-1 bank can route real incidents through.

Where we are
A working MVP. Strategist · Dynamic Runbook · Sub-Agents on MCP (Splunk, Dynatrace, AIOps, SLO Tracker, Remediation, Confluence, Jira). React builder UI, FastAPI service, SQLite state, SSE streaming. Three fix modes: suggest_only · approval_required · auto.
What's missing
Identity, RBAC, secrets, audit-grade logging, HA, model governance, eval harness, code awareness, a real onboarding flow, observability of AEGIS itself, and an operating model for scaling to 50+ apps without a hero on every incident.
North star
By Q4: ten Tier-1 services running through AEGIS in approval_required mode, MTTR cut by 40% on onboarded apps, code-aware runbooks suggesting cause-and-fix not just "restart the pod", and an onboarding flow an app team can self-serve in a week.
02 · The Roadmap

Four phases. Twelve months. Each phase earns the next.

The sequence matters. Hardening before code context — because we don't want to give a hallucinating agent more reach. Code context before scale — because onboarding the tenth app is only safe if the platform learned how to be careful by app three.

01Months 1–3

Hardening the floor

Make AEGIS something the CISO and the Auditor can sign off on before it touches another production system.

Identity & Access

  • SSO via JPMC IDAnywhere; SID for human ops, FID for service identity
  • RBAC at agent, runbook, and action granularity — separate roles for builder, approver, auditor
  • Service principals per MCP integration with least-privilege scopes
  • Just-in-time elevation for auto-mode actions

Secrets & Data

  • Migrate from .env to Vault with per-app credential rotation
  • PII redaction at the log ingestion edge — no raw customer data in agent context
  • PostgreSQL replacing SQLite; row-level security per app
  • Data classification tags carried through every agent decision

Reliability of the platform itself

  • HA deployment across two AZs; active-active orchestrator
  • Circuit breakers, retries with jitter, dead-letter queue for failed actions
  • Idempotency keys on every Sub-Agent action — no double-restarts
  • OpenTelemetry on AEGIS itself: AEGIS observes AEGIS

Model governance

  • Prompt-injection defence — input partitioning, output schema enforcement via tool use
  • Per-tenant token budgets; cost telemetry surfaced in the Ops Console
  • Model fallback chain (Sonnet → Haiku) with quality gates
  • Eval harness v1 — golden incident set + LLM-as-judge regression on every release

Audit & compliance

  • Immutable audit log of every agent decision, tool call, and fix action
  • SOX-grade approval chain for auto-mode promotions
  • 7-year retention, exportable to JPMC's e-discovery platform
  • Agent decisions reproducible from log + frozen prompts

Operational discipline

  • Blue/green deploys with shadow-traffic warm-up
  • Per-environment kill-switch reachable in under 30 seconds
  • Rollback story for every Sub-Agent action — written before the action ships
  • SRE on-call rotation for AEGIS itself
Phase 1 exit criteria AEGIS passes the JPMC TPRM-grade self-assessment. CISO sign-off on identity model. Eval harness running weekly with documented baseline. Zero reliance on developer laptops for any control-plane action.
02Months 3–5

Code context — the missing layer

Today AEGIS knows the symptoms; tomorrow it knows the source. This is the phase that turns "restart the pod" into "revert PR #234, here's why."

Repository onboarding

  • Connectors for Bitbucket Enterprise, GitHub Enterprise, GitLab
  • GitHub-App style auth with repo-scoped, time-bounded tokens
  • Initial pull at onboarding; webhook-driven incremental re-index
  • CODEOWNERS + AD resolution → who to page, who can approve

Code indexing

  • Tree-sitter parsers for Java, Python, Kotlin, Go, JS/TS
  • Symbol graph: classes, functions, call graph, import graph
  • Service topology from Spring/Quarkus annotations, K8s manifests, Terraform
  • Per-service map: repo · deployment unit · runtime · owners · runbook overrides

Embeddings & retrieval

  • Code embeddings via Voyage code-3 (or equivalent) — chunked at function boundary, not line count
  • pgvector on the same Postgres we just stood up
  • Hybrid retrieval: BM25 + dense + symbol-exact match → reranked
  • Per-repo cache for hot files; cold storage for everything else

Incident → code linking

  • Stack-frame parser → resolve to repo / file / function / line
  • Recent-commit window: what changed in the last 1h / 24h / 7d on the affected paths
  • Past-incident vector match: "we've seen this fingerprint before" with the resolution
  • Owner ping list assembled from CODEOWNERS + git-blame heuristic

Code-aware runbook synthesis

  • Strategist context now carries: alert · observability · code · diff · history
  • Output runbooks include candidate cause, the offending change, the rollback step, the owner page
  • Confidence scoring per hypothesis — runbook gates auto-execution under threshold
  • Falls back gracefully to symptom-only runbooks when code context is sparse

Learning loop

  • After every resolution: which hypothesis was right, which fix worked
  • Append to incident_history with embedding — instant memory for next time
  • Per-repo "lessons learned" prompt cache, refreshed weekly
  • Eval harness extended: did the suggested cause match the actual cause?
Phase 2 exit criteria Three pilot apps fully indexed. On the golden incident set, code-aware runbooks beat symptom-only runbooks on cause-precision by ≥40%. Hallucination rate on cause attribution under 5% (judged by app owners). Re-index latency under 5 min per push.
03Months 5–7

Self-serve onboarding

Stop being the bottleneck. Make app teams onboard themselves through a flow that earns trust before it earns autonomy.

Application profile schema

  • Single declarative spec: services, repos, owners, SLOs, criticality tier, runtime, observability bindings
  • Versioned in the app's own repo as aegis.yaml — GitOps native
  • Validates on PR; deploys through a controlled release channel

Onboarding wizard

  • Step-through UI: identity → repos → observability → runbook overrides → on-call
  • Auto-detects existing Splunk indices and Dynatrace entities by name match + suggestion
  • Pre-flight check: connectivity, permissions, sample query against each MCP
  • Generates first-pass runbook library from the team's existing Confluence pages

Trust ladder

  • Shadow (2 weeks): observes only, generates runbooks, executes nothing
  • Suggest-only: proposals on the bridge, humans drive — graduation gated by ≥85% suggestion-accept rate
  • Approval-required: AEGIS proposes and stages; humans approve, AEGIS executes
  • Auto: scoped to safe actions on Tier-3 services with one-click rollback always present

Per-app evaluation

  • Each onboarded app has its own golden set of past incidents
  • Weekly regression: did this week's runbooks beat last week's on the same incidents?
  • Owner-facing scorecard — coverage, accuracy, time saved, false-positive rate
  • App teams can promote/demote AEGIS modes from their own console
Phase 3 exit criteria Five apps onboarded without my team writing a line of integration code. Median time from "wizard start" to "first generated runbook" under 60 minutes. At least two apps graduated past Shadow into Suggest-only on their own evidence.
04Months 7–12

Scale, governance, and the operating model

Move from "ten apps where it works" to "a platform the bank treats as infrastructure, with the rituals, reporting, and economics that implies."

Cross-app intelligence

  • Anonymised pattern library across onboarded apps
  • "Apps with shared dependencies" graph — when Postgres misbehaves, who's affected?
  • Federated learning: a bad fingerprint detected in one app warns the rest
  • Reusable agents and runbook templates published to an internal marketplace

Enterprise governance

  • Quarterly model review: drift, accuracy, cost, incident counter-factual
  • Change-advisory integration — every auto-mode promotion logged as a CAB record
  • Independent eval audit by a non-AEGIS team, twice a year
  • Model cards published per agent — what it does, what it can't, how it fails

Cost & ROI economics

  • Toil hours saved per onboarded app, baselined at onboarding
  • MTTR delta vs. pre-AEGIS baseline, attributed by mode (suggest / approve / auto)
  • Token-cost per resolved incident — track and squeeze
  • Executive dashboard: business value in dollars, not just dashboards

The operating model

  • SRE platform team becomes a customer success function for AEGIS
  • Each onboarded app gets a quarterly business review — what worked, what to retire
  • "Embedded SRE" pattern for the first 30 days of every onboarding
  • AEGIS office hours, demo Fridays, internal AMA — adoption is a relationship, not a release
Phase 4 exit criteria 25+ apps onboarded across three lines of business. Provable MTTR reduction of 40%+ on tier-1 services. AEGIS surviving an independent audit. A successor on the team who can run a release cycle without me in the room.
03 · Deep Dive

Code-aware runbooks, layer by layer.

The shortest version of the case for this phase: an agent that can read the diff that broke production gives you a different category of runbook than one that can only read the alert. This is the architecture for that capability — built to slot under the existing Strategist without disturbing the demo's contract.

01
Repository onboarding
Auth · Pull · Webhook
GitHub-App style auth scoped per app. Initial clone, then incremental updates via webhooks on push, PR merge, and tag. CODEOWNERS resolved against AD on first index — gives us the human routing graph that every later layer reuses.
02
Code indexing
Tree-sitter · Symbol graph · Topology
Per language, parse to AST and emit the symbol graph: class · function · import · call. Cross-link to deployment topology pulled from K8s manifests and Terraform. Output is a graph where a node is "a thing that can fail" and edges are "things that fail because of it."
03
Embedding & vector store
Code-tuned model · pgvector · Hybrid retrieval
Chunk on function boundaries, not lines — preserves semantic units. BM25 for symbol-exact queries (find me OrderProcessor.validate); dense for semantic queries (functions that look like ours and have failed elsewhere); rerank for the final shortlist.
04
Incident-to-code resolver
Stack frames · Recent diffs · Past incidents
When the Strategist receives an alert, this layer assembles a code-context packet: the failing function source, the diff that touched it most recently, the last three PRs through the same path, and the top-3 historically-similar incidents with their resolutions. Bounded to 8K tokens — tight enough to not bloat the Strategist's window.
05
Code-aware Strategist
Confidence-scored hypotheses · Cause-and-fix runbooks
The Strategist's prompt is unchanged in shape — it just receives a richer context packet. Output now carries explicit hypotheses with confidence scores. Below threshold: degrade to symptom-only runbook. Above threshold: propose code-level fix (revert, hotfix, feature-flag toggle) as a first-class step.
06
Resolution capture & learning
Post-mortem ingestion · Memory · Eval feedback
After resolution: capture which hypothesis was correct, which fix actually worked, the timing, and the human notes. Vector-index this into incident_history — a permanent, searchable memory. Feeds the per-repo "lessons learned" prompt cache and the weekly eval regression set.
T+0s
Dynatrace fires P1 — cds-gateway latency p99 > 2.4s. Webhook hits the Ingestor.
T+3s
AIOps MCP correlates with a Splunk error spike: NullPointerException in OrderProcessor.validate() across three pods.
T+5s

T+9s

T+90s

04 · The FDE Lens

Two readings of "forward-deployed" — both apply, both compound.

The first lens is operational: AEGIS doesn't get adopted, it gets deployed — and every onboarded app team is an FDE engagement, not a ticket. The second is positional: AEGIS is the strongest portfolio asset I have for the FDE roles I'm targeting next. Treating both deliberately is what turns this from "a thing my team uses" into a category-defining piece of work.

Lens one — AEGIS as a forward-deployed product

The Palantir / Anthropic / Databricks FDE pattern is the right operating template for AEGIS adoption inside the bank. Each app team is an "engagement," not a config form.

The engagement loop

  • ›Discovery — embedded SRE shadows the team's incidents for one full on-call rotation. We learn what hurts before we offer a fix.
  • ›Onboard — together, fill in aegis.yaml. We tune retrieval, write the first runbook overrides, run pre-flight on the MCPs.
  • ›Shadow — two weeks where AEGIS observes only. We compare every generated runbook to the human response. They earn trust through evidence, not promises.
  • ›Promote — graduate through Suggest → Approve → Auto on demonstrated accuracy. The team controls the throttle.
  • ›Sustain — quarterly review, not "set and forget." We measure value created, prune what stopped earning it.

Why this matters for the platform

The single most common failure mode for an internal AI platform is the gap between the demo and the messy reality of someone else's codebase. The FDE model exists exactly to close that gap. AEGIS uses it deliberately: the platform's job is to make the engagement repeatable; the engagement's job is to keep the platform honest.

Lens two — AEGIS as career asset

The FDE roles I'm targeting (Anthropic, OpenAI, Databricks, Salesforce Agentforce, Cohere, Palantir) are looking for one thing in a portfolio: did you ship an agentic system into a real, regulated production environment, and did you make it survive? That's exactly what AEGIS is. The work is to publish it well enough that this is obvious.

The publishable surface

  • ›Architecture brief — AEGIS in Anthropic's vocabulary: an orchestrator-workers pattern with an evaluator-optimizer loop on the runbook layer. Mapped to MCP. Posted to forever_an_arjuna.
  • ›Failure-mode tree — the top ten ways AEGIS fails, the mitigation for each, the residual risk we accept. This is the artifact senior FDE interviewers care about most.
  • ›Eval harness writeup — golden set construction, judge-LLM design, drift detection. Standalone-shippable; the harness itself is a portfolio piece.
  • ›Code-context essay — the missing layer in agentic SRE. A first-principles argument that earns the right to publish the architecture above it.
  • ›FDE Operating Manual — a one-pager I keep in front of me through every interview, derived from this brief.

What stays inside the bank, what leaves

JPMC-specific telemetry, runbooks, identifiers, and any architecture detail that constitutes IP — never leave. The patterns (orchestrator-workers, evaluator-optimizer, code-context retrieval, FDE-style onboarding) — these are industry-standard concepts I've practiced at production scale. That's the line, drawn the way it has always been drawn between AEGIS and untech.ai.

06 · What could go wrong

Risks worth naming early.

Over-trust in code context
A confident-sounding wrong cause is worse than no cause. Mitigation: hard confidence thresholds, mandatory human approval below them, eval harness that explicitly tests "the agent was sure and was wrong."
Indirect prompt injection through logs and tickets
A Splunk log line containing adversarial instructions can manipulate the Strategist. Mitigation: input partitioning, structured-output enforcement, an injection-detection eval that we run before every release.
Regulatory creep
An autonomous agent that touches production in a Tier-1 bank attracts CISO, ORM, and audit attention quickly. Mitigation: bring them in at Phase 1, not Phase 4. Their concerns are not blockers; their absence is.
Adoption stalls at app three
Internal tools are graveyards of platforms that worked for the team that built them. Mitigation: the FDE operating model — explicit engagements, named owners, quarterly value reviews. Adoption is a relationship.
IP boundary leakage
AEGIS framing accidentally migrating into untech.ai or public writing in a way that creates legal ambiguity. Mitigation: the boundary I already hold — AEGIS-specific architecture stays inside; industry patterns are mine to write about.
Cost runaway under high-incident weeks
An agent run is cheap; ten thousand are not. Mitigation: per-tenant token budgets, model fallback chain, alert when daily spend crosses 1.5× rolling median.

The demo earned us the right to ask a harder question — not can it work, but can the bank lean on it. The next twelve months are the answer to that question, written in code, audits, and onboarded teams.

Every phase is sequenced so the work earns the next phase's permission. Hardening earns the right to add code context. Code context earns the right to onboard widely. Onboarding earns the right to scale governance. And underneath all of it, the FDE posture — toward our internal customers and toward the world I'm writing this for — is the operating discipline that keeps it honest.
