AEGIS · Strategic Roadmap

From MVP to Production AIOps

AEGIS (Automated Engine for Guided Incident Strategy) has a working MVP: trigger → gather intelligence → strategist generates runbook → basic execution. This document maps every module needed for a production-grade AIOps solution and sequences them into a phased roadmap with measurable milestones.

✓ What the MVP Proved

MCP connectivity works — Splunk, Dynatrace, Confluence, Jira all reachable via MCP servers on VDI.
Parallel intelligence gathering works — all 4 tools queried concurrently, real production data returned.
Claude generates usable runbooks — the Strategist produces YAML runbooks from real signals that SREs recognize as sensible.
Basic UI works — trigger incident, watch intelligence gather, see generated runbook.

✗ What's Missing for Production Value

No orchestration logic — runbook steps don't actually execute against tools, no approval gates, no branch logic.
No alert correlation — system reacts to one trigger at a time, can't group related alerts or suppress noise.
No topology awareness — doesn't know CDS gateway depends on Pithos depends on IBM CM. Can't trace blast radius.
No historical learning — every incident starts from zero. No memory of past incidents, resolutions, or patterns.
No proactive detection — purely reactive. Only runs when manually triggered, not when anomalies emerge.
No change correlation — doesn't automatically link deployments/config changes to emerging issues.
No metric tracking — no MTTR measurement, no cost tracking, no quality scoring of generated runbooks.
No audit trail — can't prove what the system did, why, and who approved it.

The Gap in One Sentence

The MVP can diagnose and plan for a single, manually-triggered incident using real data. A production AIOps solution must detect, correlate, diagnose, plan, execute, verify, learn, and improve across continuous alert streams — autonomously for known patterns, human-gated for novel ones.

Industry Benchmark

Top-tier AIOps platforms (BigPanda, Datadog Event Management, ServiceNow ITOM) deliver these core capabilities:

1. Alert noise reduction — 90-99% reduction via correlation and deduplication. We have: 0%.
2. Topology-aware RCA — trace causality through service dependency graphs. We have: flat correlation only.
3. Change impact analysis — auto-link deployments/config changes to emerging incidents. We have: manual.
4. Predictive alerting — detect anomalies before they breach thresholds. We have: nothing.
5. Automated remediation — closed-loop fix for known patterns. We have: mock only.
6. Continuous learning — improve from every resolved incident. We have: nothing.
7. Audit + compliance — full trail of every automated action. We have: nothing.

The 9 Domains That Make a Complete AIOps Solution

Each domain is a coherent set of capabilities. Some are foundational (must build first), some are differentiators (where AEGIS becomes uniquely valuable), and some are table-stakes (needed for production but not exciting). The order matters.

M1 Orchestration & Execution Engine
FOUNDATIONAL HIGHEST PRIORITY

The "Hands" — Make the generated runbook actually DO things

Current state: Runbook is generated but steps don't execute. No approval gates. No branching.

DAG Executor

Parse runbook YAML → build dependency graph → execute steps respecting order, parallel groups, and branch logic. Resume from checkpoint on failure.

must-have~5 days

Typed Sub-Agents

DiagnosticAgent (read-only), DecisionAgent (LLM reasoning), RemediationAgent (write, always gated). Permission boundaries enforced in code.

must-have~3 days

Human Approval Gate

Pause execution at remediation steps. Show full diagnosis context. Approve/reject/modify. Timeout → auto-escalate. Web + Slack.

must-have~3 days

Verification Loop

After remediation: wait, re-check metrics, confirm recovery. If not recovered → rollback or escalate. Closed-loop automation.

must-have~2 days
M2 Alert Correlation & Noise Reduction
DIFFERENTIATOR

Go from single-trigger to continuous alert processing

Current state: One incident = one manual trigger. No awareness of related alerts.

Alert Ingestion Pipeline

Webhook receivers for Dynatrace, PagerDuty, Splunk alerts. Normalize all alerts into a common event schema with timestamp, source, severity, affected entity, and raw payload.

high-value~3 days

Time-Window Correlation

Group alerts that fire within a configurable window (e.g., 5 min) for the same service or related services. Reduce N alerts → 1 incident. Start with rule-based, graduate to ML.

high-value~4 days

Deduplication Engine

Identical alerts within a window collapse to a single event with count. Fingerprint-based: hash(source + entity + error_signature) → dedupe key.

high-value~2 days

LLM-Powered Grouping

For alerts that aren't identical but semantically related: use embeddings to cluster. "DB connection timeout" + "API 504 gateway timeout" → same incident.

advanced~4 days
M3 Service Topology & Blast Radius
DIFFERENTIATOR

Understand what depends on what — and who gets hurt

Current state: No dependency awareness. Can't trace upstream/downstream impact.

Service Dependency Graph

Build a graph of service dependencies from Dynatrace topology, Confluence architecture docs, and manual config. Store in a queryable format (NetworkX or Neo4j-lite).

high-value~4 days

Blast Radius Calculator

Given a failing service, walk the dependency graph to identify all downstream consumers. Score severity by number and criticality of affected consumers.

high-value~2 days

Topology-Aware Diagnosis

Feed dependency context into Strategist. "CDS gateway depends on Pithos. Pithos is also alerting → investigate Pithos first." Changes diagnosis strategy.

high-value~2 days

Auto-Discovery via OTel

As OTel maturity improves, auto-build topology from trace data. Service A calls Service B → edge in the graph. No manual config needed.

advanced~5 days
M4 Knowledge Base & Institutional Memory
DIFFERENTIATOR MOAT

Learn from every incident — the system gets smarter over time

Current state: Every incident starts from scratch. No memory of past resolutions.

Incident Vector Store

Every resolved run (incident context + diagnosis + runbook + outcome) is embedded and stored in a vector DB (pgvector or ChromaDB). Enables semantic search: "show me incidents similar to this one."

high-value~4 days

RAG-Powered Strategist

Before generating a new runbook, retrieve the 3-5 most similar past incidents. Feed their runbooks and outcomes into the Strategist context. "Last time this happened, rollback worked. The time before, it was an upstream issue."

high-value~3 days

Runbook Effectiveness Scoring

After each run: did the runbook resolve the issue? How long did it take? Was the root cause correct? Score and store. Over time, the Strategist learns which patterns work and which don't.

high-value~3 days

Confluence Knowledge Indexer

Index existing Confluence runbooks, architecture docs, and postmortems into the vector store. The Strategist can cite specific Confluence pages in its reasoning.

advanced~3 days
M5 Change Intelligence
HIGH IMPACT

Link deployments and config changes to emerging incidents

Current state: Jira data fetched but not correlated to incident timing automatically.

Change Event Tracker

Continuously ingest deployment events, config pushes, feature flag toggles, and infra changes from Jira, CI/CD, and ServiceNow. Maintain a timeline.

high-value~3 days

Change-Incident Correlator

"Incident started at 14:18. Deployment build-4891 pushed at 14:15. Correlation: 0.92." Auto-inject into Strategist context when temporal proximity is high.

high-value~3 days

Deployment Risk Scorer

Before deployment: score risk based on change size, service criticality, time of day, historical failure rate of similar changes. "This deploy is high-risk: large diff + Friday afternoon + critical service."

advanced~4 days
M6 Proactive Detection & Anomaly Intelligence
ADVANCED

Detect problems before they become incidents

Current state: Purely reactive. Only runs on manual trigger or alert webhook.

Scheduled Health Scans

Cron-based: every 15 min, pull key metrics from Dynatrace + Splunk for all 12 services. Compare against baselines. If anomalous → auto-trigger AEGIS investigation (not remediation).

high-value~4 days

Exception Pattern Monitor

Daily scan of Splunk error logs. Categorize as known vs. novel via Drain3 log template extraction. Surface new exception patterns before they escalate.

high-value~5 days

SLO Burn Rate Alerting

Continuous monitoring of error budget burn rates. Multi-window: 1h, 6h, 24h. Trigger AEGIS at Warning level, not just Critical — catch problems earlier.

high-value~3 days

Predictive Anomaly Detection

ML models (LSTM or Prophet) on key metrics to predict threshold breaches 1-4 hours ahead. "Error rate will breach SLO in ~2 hours at current trend."

advanced~8 days
M7 Operations Dashboard & Workflow UX
ADOPTION

The UI that makes the team USE this every day

Current state: Basic single-page trigger + result view. No history, no search, no team workflow.

Real-Time Execution Dashboard

React app. Live run trace with step-by-step execution, approval queue, status indicators. WebSocket-driven. The war-room view.

must-have~5 days

Run History & Search

Paginated history of all runs. Filter by app, env, status, time range, root cause category. Full audit trail per run.

must-have~3 days

Runbook Diff & Version View

Compare generated runbooks across runs for the same service. See how AEGIS's plans evolve as it learns. Monaco editor with YAML syntax highlighting.

high-value~2 days

Slack/Teams Integration

Post diagnosis summaries to incident channels. Approve remediation from Slack. Get @mentioned when approval is needed.

high-value~4 days

MTTR & Cost Analytics

Track per-incident: time to diagnose, time to plan, time to resolve. LLM cost per run. Show trends over time. This is your ROI dashboard for leadership.

high-value~3 days
M8 Safety, Audit & Governance
TABLE STAKES JPMC REQUIRED

What makes this deployable at a bank

Current state: No audit trail. No RBAC. No compliance artifacts.

Structured Audit Logger

JSON log for every: trigger, MCP call, LLM call (with tokens + cost), approval decision (who/when/what), remediation execution, state transition. Immutable. Queryable.

must-have~3 days

RBAC (Role-Based Approval)

Junior SRE: approve restarts. Senior SRE: approve rollbacks. Manager: approve scaling changes. Role definitions configurable per team.

high-value~4 days

Confidence-Based Auto-Approve

If same runbook pattern approved 20+ times with 100% success → offer auto-approve with countdown. Builds trust incrementally. Configurable per action type.

advanced~3 days

Demo Mode / Dry Run

Run full pipeline but mock all write operations. For testing, demos, and shadow-mode pilots alongside manual SRE response.

must-have~1 day
M9 Infrastructure & Reliability
PLUMBING

Make the system itself reliable

Current state: If it crashes, restart. No persistence. No retry. No scaling.

State Persistence + Crash Recovery

SQLite → PostgreSQL. Checkpoint after every step. Resume interrupted runs from last completed step.

must-have~3 days

Error Handling & Retry

Per-step retry with exponential backoff. MCP timeout → skip + log + continue. LLM failure → Opus fallback. Graceful degradation everywhere.

must-have~3 days

Context Accumulator Hardening

3-tier summarization (raw → extractive → abstractive). Token budget enforcement. Prevent 50KB Splunk responses from blowing the context window.

must-have~2 days

Docker / GKP Deployment

Containerize for Gaia/CPSR deployment. Docker Compose for dev, Helm chart for GKP prod. Health endpoints for K8s probes.

high-value~3 days

Phased Roadmap — 4 Horizons

Each horizon produces a deployable milestone that delivers measurable value. Not everything needs to ship for AEGIS to be useful — the roadmap is sequenced so that each phase stands alone as a usable product, and each subsequent phase compounds value.

H0 · DONE

MVP — Proof of Concept

Completed
MCP connectivity to 4 tools. Parallel intelligence gathering. Strategist generates YAML runbook from real data. Basic UI to trigger and view results. Proved the core hypothesis: LLMs can generate actionable SRE runbooks from production data.
✓ MCP Layer ✓ Strategist (basic) ✓ Basic UI

Closed-Loop Execution — The system actually DOES things

~6 weeks
The goal: Go from "generates a plan" to "executes the plan end-to-end with human approval gates." This is the L1 → L2 maturity jump. After H1, AEGIS reduces MTTR by automating the execution loop, not just the diagnosis.

What ships:
• DAG executor with typed sub-agents and permission boundaries
• Web-based approval gate with full diagnosis context
• Verification loop (post-remediation health check)
• Change-incident correlation (link deploys to alerts)
• Alert webhook ingestion (Dynatrace/PagerDuty → auto-trigger)
• Run history with search + audit trail
• MTTR tracking dashboard
• State persistence + crash recovery
• Error handling + retry logic
• Demo mode (mock writes) + dry-run support
M1 Orchestration (full) M5 Change Correlation (basic) M7 Dashboard (core) M8 Audit + Demo Mode M9 Persistence + Retry
Target MTTR reduction: 30-40% Runbook execution: fully automated Write safety: 100% human-gated Pilot: 1 service (shadow mode)
H2 · LEARN

Intelligence Layer — The system gets smarter over time

~6 weeks
The goal: Every resolved incident makes AEGIS better at handling the next one. This is what creates the competitive moat — the knowledge base is unique to your org.

What ships:
• Incident vector store (embed + store every resolution)
• RAG-powered Strategist (retrieve similar past incidents before planning)
• Runbook effectiveness scoring (track which patterns work)
• Alert deduplication + time-window correlation
• Service dependency graph (manual + Dynatrace topology)
• Blast radius calculator
• Topology-aware diagnosis ("check upstream first")
• Confluence knowledge indexer
• Slack/Teams integration for approvals
• RBAC (role-based approval tiers)
M4 Knowledge Base (full) M2 Alert Correlation (basic) M3 Topology (manual + DT) M7 Slack + RBAC
Diagnosis accuracy: improves with volume Alert noise reduction: 70-80% Knowledge base: 100+ indexed incidents Pilot: 3-5 services
H3 · PREDICT

Proactive Operations — Catch problems before they're incidents

~6 weeks
The goal: Shift from reactive incident response to proactive health management. AEGIS doesn't wait for alerts — it continuously monitors and surfaces emerging issues.

What ships:
• Scheduled health scans (every 15 min across all services)
• Exception pattern monitor (daily Splunk scan, known vs. novel)
• SLO burn rate alerting (multi-window: 1h/6h/24h)
• LLM-powered alert grouping (semantic similarity clustering)
• Auto-discovery topology via OTel trace data
• Confidence-based auto-approve for known patterns
• Deployment risk scorer
• Predictive anomaly detection (trend-based threshold breach forecasting)

The Strategic Insight

H1 (Execute) is where AEGIS goes from "interesting demo" to "tool my team uses daily." It's the only horizon that's purely about making the existing capability operational.

H2 (Learn) is where AEGIS becomes a moat. The knowledge base is unique to your org — no off-the-shelf tool has your incident history, your failure patterns, your topology. Every incident makes the system smarter. This is the compounding flywheel.

H3 (Predict) is where AEGIS shifts the paradigm from reactive to proactive. This is the "Level 3" on the SRE AI Maturity Model you're pitching to Kunal. Most orgs never get here. If yours does, it's a competitive advantage in operational quality.
